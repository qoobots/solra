## 一、技术架构总纲

### 1.1 技术架构定位

`07索拉应用架构设计.md` 回答"系统分哪些子项目、服务怎么拆、怎么通信"（**What**），本文档回答"用什么语言/框架/工具实现、代码怎么组织、工程怎么管"（**How**）。

```
文档层级关系：
┌─────────────────────────────────────────────────────────┐
│  06 功能清单进度    → 项目→子项目→模块→功能 四层结构      │
│  07 应用架构设计    → DDD + 微服务 + 事件驱动 + CQRS      │
│  08 技术架构设计    → 技术栈选型 + 工程结构 + 开发规范     │  ← 本文档
└─────────────────────────────────────────────────────────┘
```

### 1.2 技术选型原则

| 原则 | 说明 | 反例 |
|------|------|------|
| **成熟优于新奇** | 优先选择有大规模生产案例的技术 | 盲目追新导致踩坑 |
| **统一优于多样** | 跨子项目统一语言/框架/工具链，降低认知成本 | 每个子项目各自选型 |
| **渐进复杂度** | 技术验证期用最简单的方案，规模验证期再引入复杂方案 | 起步就上分布式事务 |
| **端边云一致** | 端侧和云侧尽可能共享代码/协议/类型定义 | 端云模型定义不一致 |
| **可替换性** | 每项技术选择保留替换路径，避免供应商锁定 | 深度绑定某个云厂商API |

## 二、技术栈详细选型

### 2.1 编程语言选型

| 平台/服务 | 语言 | 版本 | 选型理由 |
|-----------|------|------|---------|
| **跨平台核心引擎（Solra Core）** | C++ | 17 | 性能关键路径（渲染/推理/编解码），iOS/Android/Desktop共享代码 |
| **云端业务服务** | Java | 17 LTS | 生态成熟、招聘市场大、Spring Boot 3.x 原生支持 GraalVM |
| **AI推理服务** | Python | 3.11 | AI/ML生态首选，vLLM/PyTorch/HuggingFace全支持 |
| **iOS App** | Swift | 5.9+ | Apple生态一等公民，SwiftUI提供声明式UI |
| **Android App** | Kotlin | 2.0+ | Google官方推荐，Compose实现声明式UI |
| **Web前端 (H5)** | TypeScript | 5.x | 类型安全 + React生态 |
| **API网关配置** | Lua | 5.1 | APISIX插件开发语言 |
| **CI/CD脚本** | Python / Shell | - | Glue code，与AI服务统一Python |
| **基础设施即代码** | HCL (Terraform) | - | 多云IaC标准 |
| **Protobuf/IDL** | Protocol Buffers | proto3 | 跨语言服务契约，生成代码 |

### 2.2 框架与运行时

| 层次 | 技术 | 版本 | 说明 |
|------|------|------|------|
| **业务微服务框架** | Spring Boot 3.x | 3.2+ | 生产验证充分，生态完善 |
| | Spring Cloud (可选) | 2023.0+ | 仅服务发现/配置管理，避免重量级组件 |
| **AI推理服务框架** | FastAPI | 0.110+ | 异步高性能，原生支持流式响应 |
| **前端框架** | React | 18.x | 生态最大，H5/管理后台统一 |
| **移动端声明式UI** | SwiftUI / Compose | 系统原生 | 平台原生UI框架，避免跨平台UI方案 |
| **3D渲染（iOS/Android）** | Metal 3 / Vulkan 1.3 | 系统原生 | 极致性能，避免OpenGL ES的中间损耗 |

| **gRPC** | gRPC | 1.60+ | 微服务间高性能通信 |
| **序列化** | Protocol Buffers | proto3 | 跨语言类型一致，比JSON体积小80% |

### 3.3 编码规范

## 九、CI/CD 技术实现

### 9.1 构建流水线细节

```yaml
# .github/workflows/ci-pr.yml (概念示例)
name: PR Check
on:
  pull_request:
    branches: [master]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Java Checkstyle
        run: ./gradlew checkstyleMain
      - name: Python Lint
        run: poetry run ruff check ai-services/
      - name: Proto Lint
        run: buf lint contracts/proto/

  test:
    runs-on: ubuntu-latest
    steps:
      - name: Java Unit Tests
        run: ./gradlew test jacocoTestReport
      - name: Python Tests
        run: poetry run pytest ai-services/
      - name: C++ Tests
        run: cd core && cmake --build build --target test

  security:
    runs-on: ubuntu-latest
    steps:
      - name: Dependency Scan
        run: ./gradlew dependencyCheckAnalyze
      - name: SAST
        uses: sonarsource/sonarcloud-github-action@v2
      - name: Container Scan
        run: trivy image solra/*:latest

  quality-gate:
    needs: [lint, test, security]
    runs-on: ubuntu-latest
    steps:
      - name: Coverage Gate (>80%)
        run: ./scripts/check-coverage.sh --threshold 80
      - name: Performance Baseline
        run: ./scripts/check-perf-baseline.sh
```

### 9.2 部署策略

| 环境 | 部署方式 | 验证方式 | 回滚时间 |
|------|---------|---------|---------|
| **Dev** | Git Push → Auto Deploy | 冒烟测试 | <2分钟 |
| **Staging** | PR Merge → Auto Deploy | 完整回归测试 | <2分钟 |
| **Production** | 手动审批 → 灰度发布 (5%→20%→50%→100%) | 渐进验证+自动回滚 | <2分钟 |
| **Hotfix** | 紧急通道 → Staging验证 → 全量发布 | 冒烟测试 | <2分钟 |

## 十、技术风险与缓解

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| **GPU推理成本过高** | 虚拟人体验降级 / 亏损 | 高 | 端侧SLM分担>50%请求；推理批处理提高GPU利用率；按虚拟人分级（付费优先GPU） |
| **端侧3D性能不足** | 低端机体验差 / 用户流失 | 中 | LOD四级降级；低端机降级为"轻3D"模式；性能分级运行（旗舰/中端/低端） |
| **空间加载超时** | 用户跳出率高 | 中 | 预测性预加载；CDN边缘节点；P2P分发（技术验证期后） |
| **WebRTC连通率低** | 多人社交体验受损 | 中 | TURN中继服务器兜底；UDP→TCP降级；国内BGP多线接入 |
| **内容安全漏审** | 监管风险 / 下架 | 低 | 人机双重审核（机器初筛+人工复核）；安全模型持续迭代 |
| **分布式一致性问题** | 数据不一致 / 用户投诉 | 低 | 最终一致性设计；事件溯源+补偿机制；对账系统 |
| **供应商锁定** | 迁移成本高 | 中 | 多云架构（阿里云+腾讯云+自建）；接口抽象层不绑定云厂商API |
| **核心人才流失** | 研发进度受阻 | 低 | 文档完备；Code Review知识共享；关键模块至少两人掌握 |
