## 3. 分表方案

### 3.1 按时间分区（PostgreSQL Partitioning）

**适用表**：messages、conversations、space_visits、notifications、audit_logs

```sql
-- 创建分区表
CREATE TABLE messages (
    id UUID NOT NULL,
    conversation_id UUID NOT NULL,
    role VARCHAR(20) NOT NULL,
    content TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);

-- 创建月度分区
CREATE TABLE messages_2026_07 PARTITION OF messages
    FOR VALUES FROM ('2026-07-01') TO ('2026-08-01');

CREATE TABLE messages_2026_08 PARTITION OF messages
    FOR VALUES FROM ('2026-08-01') TO ('2026-09-01');

-- 自动创建分区（pg_partman扩展）
SELECT partman.create_parent(
    p_parent_table := 'public.messages',
    p_control := 'created_at',
    p_type := 'native',
    p_interval := '1 month',
    p_premake := 3  -- 提前创建3个月
);
```

### 3.2 分区管理策略

| 操作 | 策略 | 工具 |
|------|------|------|
| 创建分区 | 提前3个月自动创建 | pg_partman |
| 归档 | 6个月以上数据迁移到归档表 | pg_partman |
| 清理 | 12个月以上归档表删除 | 定时任务 |
| 维护 | 每月运行VACUUM ANALYZE | cron |

### 3.3 分区查询优化

```sql
-- 查询自动分区裁剪（constraint_exclusion = partition）
SELECT * FROM messages
WHERE created_at >= '2026-07-01'
  AND created_at < '2026-08-01';  -- 仅扫描 messages_2026_07 分区

-- 跨分区查询使用并行扫描
SET max_parallel_workers_per_gather = 4;
```

## 4. 数据归档策略

### 4.1 冷热数据分离

```
热数据 (0-3月)   → 主库 SSD → 高频访问
温数据 (3-6月)   → 主库 HDD → 低频访问
冷数据 (6-12月)  → 归档库    → 按需查询
冰数据 (>12月)   → 对象存储  → 极少访问(需恢复)
```

### 4.2 归档流程

```sql
-- 1. 将旧分区从主表分离
ALTER TABLE messages DETACH PARTITION messages_2025_12;

-- 2. 将分区附加到归档表
ALTER TABLE messages_archive ATTACH PARTITION messages_2025_12
    FOR VALUES FROM ('2025-12-01') TO ('2026-01-01');

-- 3. 可选：导出到对象存储
COPY messages_2025_12 TO '/archive/messages_2025_12.csv' CSV;
```

### 4.3 定时任务

```yaml
# K8s CronJob
apiVersion: batch/v1
kind: CronJob
spec:
  schedule: "0 3 1 * *"  # 每月1日凌晨3点
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: archive-job
            image: solra/data-archiver:latest
            command: ["python", "archive.py", "--months", "6"]
```

## 5. 分片键设计

### 5.1 对于未来可能需要水平分片的表

| 表 | 分片键 | 分片算法 | 预估分片数 | 触发条件 |
|----|--------|----------|-----------|----------|
| users | id | 一致性哈希 | 4→16 | >5000万用户 |
| spaces | id | 一致性哈希 | 4→8 | >1000万空间 |
| orders | order_no | 取模 | 4→8 | >1亿订单 |

### 5.2 分片中间件

**推荐方案**：ShardingSphere-Proxy

```yaml
# ShardingSphere配置示例
rules:
- !SHARDING
  tables:
    users:
      actualDataNodes: user_db_${0..3}.users
      databaseStrategy:
        standard:
          shardingColumn: id
          shardingAlgorithmName: user_hash
  shardingAlgorithms:
    user_hash:
      type: HASH_MOD
      props:
        sharding-count: 4
```

> **注意**：当前阶段数据量尚未达到需要水平分片的规模，分表+分区可满足3年内需求。分片方案作为预留设计，不在此阶段实施。

## 6. 全局唯一ID

### 6.1 ID生成方案

| 方案 | 说明 | 使用场景 |
|------|------|----------|
| UUID v7 | 时间有序UUID | 主键默认方案 |
| Snowflake | 分布式自增ID | 订单号等业务ID |

### 6.2 UUID v7

```
格式: 时间戳(48bit) + 版本(4bit) + 随机(12bit) + 变体(2bit) + 随机(62bit)
示例: 018f3a5c-9b00-7e00-a000-000000000000

优点: 时间有序，索引友好，无碰撞风险，无需中心化生成
```

### 6.3 Snowflake（订单号）

```
格式: 时间戳(41bit) + 机器ID(10bit) + 序列号(12bit)
示例: 7182039485734912

PostgreSQL实现:
CREATE SEQUENCE order_no_seq START 10000000;
```

## 7. 跨分片查询

### 7.1 禁止的操作

| 操作 | 原因 |
|------|------|
| 跨分片JOIN | 性能不可控 |
| 跨分片事务 | 分布式事务复杂 |
| 跨分片聚合 | 需中间件支持 |

### 7.2 替代方案

| 需求 | 方案 |
|------|------|
| 跨服务数据聚合 | API组合 / CQRS读模型 |
| 全局搜索 | Elasticsearch |
| 全局统计 | ClickHouse / 离线数仓 |
| 关联查询 | 冗余必要字段，避免JOIN |
