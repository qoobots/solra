## 1. 可用性目标

### 1.1 分级SLA

| 服务等级 | 目标可用性 | 允许年宕机 | 涉及服务 |
|----------|-----------|-----------|----------|

| 场景 | RTO | RPO |

## 2. 高可用架构

### 2.1 多层冗余设计

```
           ┌─────────────────────────┐
           │     DNS (Geo Failover)  │  ← 智能DNS
           └───────────┬─────────────┘
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
   ┌─────────┐   ┌─────────┐   ┌─────────┐
   │Region A │   │Region B │   │Region C │   ← 多区域
   │ (Active)│   │ (Active)│   │(Standby)│
   └────┬────┘   └────┬────┘   └────┬────┘
        │              │              │
   ┌────┴────┐   ┌────┴────┐   ┌────┴────┐
   │AZ1  AZ2 │   │AZ1  AZ2 │   │   AZ1   │  ← 多可用区
   │(双活)   │   │(双活)   │   │         │
   └─────────┘   └─────────┘   └─────────┘
```

### 2.2 应用层高可用

| 机制 | 实现 |
|------|------|
| 多副本部署 | 核心服务≥3副本，跨AZ分布 |
| 健康检查 | Liveness + Readiness Probe |
| 自动扩缩 | HPA (CPU > 70%触发) |
| 优雅关闭 | Graceful Shutdown (30s drain) |
| 启动预热 | Startup Probe (60s initial) |
| Pod反亲和 | preferredDuringScheduling (跨节点) |

### 2.3 数据层高可用

| 组件 | 高可用方案 | 故障切换 |
|------|-----------|----------|
| PostgreSQL | Patroni (1主2从) + etcd | 自动Failover <30s |
| Redis | Cluster (3主3从) | 自动Failover |
| Kafka | KRaft (3节点) | Controller选举 |
| MinIO | Distributed EC | 容忍2节点故障 |
| Milvus | Distributed 多副本 | 组件级冗余 |
| Elasticsearch | 3节点集群 | 主节点选举 |
| Consul | 3节点集群 | Raft选举 |
| Nacos | 3节点集群 | Raft选举 |

## 3. 容灾设计

### 3.1 灾备等级

| 等级 | 描述 | RTO | RPO | 适用 |
|------|------|-----|-----|------|
| 热备 | 多Region Active-Active | <5min | 0 | Tier 1 |
| 温备 | 备Region服务待命 | <30min | <1min | Tier 2 |
| 冷备 | 定期备份恢复 | <4h | <1h | Tier 3 |

### 3.2 跨区域数据同步

| 数据 | 同步方式 | 延迟 |
|------|----------|------|
| PostgreSQL | 异步流复制（跨Region） | <1s |
| Redis | Redis Enterprise Active-Active | <10ms |
| Kafka | MirrorMaker 2 | <5s |
| MinIO | Site Replication | <5s |

### 3.3 灾难恢复流程

```
1. 检测故障（监控告警）
2. 确认灾难级别（单AZ/单Region/全集群）
3. 决策切换（自动/手动）
4. DNS切换流量到备用Region
5. 验证备用Region服务正常
6. 通知干系人
7. 故障Region恢复后数据回切
8. 复盘总结
```

## 5. 熔断与降级

### 5.1 熔断策略

```yaml
# Resilience4j 配置
resilience4j:
  circuitbreaker:
    instances:
      default:
        slidingWindowSize: 10
        failureRateThreshold: 50    # 失败率>50%熔断
        waitDurationInOpenState: 30s # 30秒后半开
        permittedNumberOfCallsInHalfOpenState: 3
```

### 5.2 降级方案

| 服务 | 降级场景 | 降级行为 |
|------|----------|----------|
| AVT | AI推理不可用 | 返回预设回复 |
| SPC | 推荐服务不可用 | 返回热门空间列表 |
| SOC | 实时消息不可用 | 降级为轮询模式 |
| MON | 支付网关不可用 | 展示维护公告 |
| SAF | 内容审核不可用 | 先发布后审核 |

### 5.3 限流策略

| 层级 | 限制 | 说明 |
|------|------|------|
| 网关 | 10000 req/s (全局) | Kong Rate Limiting |
| 用户 | 100 req/s | 基于用户ID |
| IP | 500 req/s | 基于IP |
| API | 按API定义 | 核心API优先保障 |

## 6. 监控与告警

### 6.1 告警规则

| 级别 | 条件 | 通知方式 | 响应时间 |
|------|------|----------|----------|
| P0 紧急 | 核心服务不可用/数据丢失 | 电话+短信+IM | 5分钟 |
| P1 严重 | 服务降级/延迟>5s | 短信+IM | 15分钟 |
| P2 警告 | 错误率>5%/磁盘>80% | IM | 1小时 |
| P3 提示 | 非关键指标异常 | 邮件 | 24小时 |

### 6.2 关键监控指标

| 指标 | 阈值 | 级别 |
|------|------|------|
| API可用性 | <99.99% | P0 |
| P99延迟 | >5s | P1 |
| 错误率 | >5% | P2 |
| CPU使用率 | >90% | P2 |
| 内存使用率 | >90% | P2 |
| 磁盘使用率 | >85% | P2 |
