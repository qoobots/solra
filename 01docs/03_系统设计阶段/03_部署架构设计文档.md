## 1. 部署架构概述

Solra 采用多云+混合部署策略：核心服务部署在私有云/公有云Kubernetes集群，AI推理使用GPU专用集群，边缘CDN加速静态内容分发。

### 1.1 部署层次

```
L0 客户端层: iOS/Android/Web/Desktop/MiniProgram
     │
L1 接入层:   CDN + API Gateway (Kong/APISIX) + WAF
     │
L2 业务层:   K8s Cluster (Spring Boot微服务)
     │
L3 AI推理层: GPU Cluster (vLLM/Triton/TensorRT)
     │
L4 数据层:   PostgreSQL + Redis + Kafka + MinIO + Milvus + Elasticsearch
```

## 3. Kubernetes集群设计

### 3.1 集群规划

| 集群名称 | 用途 | 节点规格 | 节点数(起始) | 区域 |
|----------|------|----------|-------------|------|
| solra-cn-prod | 生产业务集群 | 16C64G | 12 | CN |
| solra-cn-gpu | GPU推理集群 | 8C32G+A100×2 | 4 | CN |
| solra-cn-data | 有状态数据服务 | 32C128G+NVMe | 6 | CN |
| solra-sg-prod | 东南亚业务集群 | 16C64G | 6 | SG |
| solra-staging | 预发布环境 | 8C32G | 4 | CN |
| solra-dev | 开发测试环境 | 8C32G | 4 | CN |

### 3.2 命名空间划分

| 命名空间 | 内容 | 资源配额 |
|----------|------|----------|
| solra-system | 基础设施（Gateway/监控/日志/服务网格） | 专用 |
| solra-auth | AUTH认证服务 | 4C8G×3 |
| solra-avt | AVT虚拟人服务 | 8C16G×6 |
| solra-spc | SPC空间消费服务 | 8C16G×6 |
| solra-crt | CRT空间创建服务 | 4C8G×3 |
| solra-soc | SOC社交服务 | 4C8G×3 |
| solra-grw | GRW增长服务 | 2C4G×3 |
| solra-mon | MON变现服务 | 4C8G×3 |
| solra-saf | SAF安全服务 | 4C8G×3 |
| solra-not | NOT通知服务 | 2C4G×2 |
| solra-inf | INF基础设施 | 2C4G×2 |

### 3.3 Pod部署策略

| 策略 | 配置 |
|------|------|
| 副本数 | 核心服务≥3，非核心≥2 |
| 反亲和性 | preferredDuringScheduling (优先分布在不同节点) |
| 资源限制 | request=limit (Guaranteed QoS) |
| 健康检查 | livenessProbe (30s) + readinessProbe (10s) |
| 滚动更新 | maxSurge=1, maxUnavailable=0 |
| HPA | CPU>70%触发扩容，最大10副本 |

## 5. 数据层部署

### 5.1 PostgreSQL

| 配置项 | 值 |
|--------|-----|
| 部署模式 | Patroni HA (1主2从) |
| 版本 | PostgreSQL 16 |
| 连接池 | PgBouncer (事务模式) |
| 备份 | 每日全量 + WAL持续归档 |
| 备份保留 | 30天 |

### 5.2 Redis

| 配置项 | 值 |
|--------|-----|
| 部署模式 | Redis Cluster (6节点，3主3从) |
| 版本 | Redis 7.2 |
| 持久化 | AOF everysec + RDB每日 |
| 用途 | Session/缓存/Presence/Rate Limit |

### 5.3 Kafka

| 配置项 | 值 |
|--------|-----|
| 部署模式 | KRaft模式 (3节点) |
| 版本 | Kafka 3.7 |
| 分区数 | 每Topic默认12分区 |
| 保留策略 | 7天或1TB |

### 5.4 MinIO

| 配置项 | 值 |
|--------|-----|
| 部署模式 | Distributed Mode (4节点，每节点4盘) |
| 纠删码 | EC:4 |
| 总容量 | 64TB可用 |

### 5.5 Milvus

| 配置项 | 值 |
|--------|-----|
| 部署模式 | Distributed (2 proxy + 4 data + 2 index + 2 coord) |
| 版本 | Milvus 2.4 |
| 向量维度 | 1536 (text-embedding-3-small) |

## 6. 网络设计

### 6.1 网络分层

```
┌─────────────────────────────────────┐
│           Public Internet           │
└───────────────┬─────────────────────┘
                │
┌───────────────┴─────────────────────┐
│         DMZ (公网可达)              │
│  CDN / WAF / LB / API Gateway      │
└───────────────┬─────────────────────┘
                │
┌───────────────┴─────────────────────┐
│       Application Subnet (私网)     │
│  K8s Worker Nodes / Service Mesh   │
└───────────────┬─────────────────────┘
                │
┌───────────────┴─────────────────────┐
│        Data Subnet (私网隔离)        │
│  PostgreSQL / Redis / Kafka / MinIO │
└─────────────────────────────────────┘
```

### 6.2 网络安全

| 层级 | 措施 |
|------|------|
| DDoS防护 | 云厂商高防IP + WAF |
| 传输加密 | TLS 1.3 (所有公网流量) |
| 内部通信 | mTLS (Istio服务网格) |
| 网络策略 | K8s NetworkPolicy (零信任) |
| 防火墙 | 安全组最小开放原则 |

## 7. CI/CD流水线

```
Git Push → GitHub Actions
  1. Lint + Test
  2. Build Image (多架构: amd64/arm64)
  3. Push to Harbor Registry
  4. Deploy to dev (自动)
  5. Deploy to staging (手动触发)
  6. Deploy to prod (审批+灰度)
```

### 7.1 灰度发布策略

```
100% → canary 5% (5min观察)
    → canary 20% (10min观察)
    → canary 50% (15min观察)
    → 100% (全量)
    → 异常自动回滚
```
