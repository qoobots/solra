## 1. 安全设计原则

| 原则 | 说明 |
|------|------|
| **纵深防御** | 多层安全防护，不依赖单一措施 |
| **最小权限** | 每个组件仅拥有完成任务所需最小权限 |
| **零信任** | 不信任任何网络边界，所有请求需认证 |
| **安全左移** | 安全融入开发全流程（DevSecOps） |
| **默认安全** | 默认配置即安全，需主动降低安全性 |
| **隐私设计** | 从设计阶段嵌入隐私保护 |

## 3. 认证与授权

### 3.1 认证方案

```
用户 → 登录请求
  1. 账号密码/第三方OAuth/生物识别
  2. AUTH Service验证
  3. 签发 JWT (RS256签名)
     - Access Token: 15min
     - Refresh Token: 7d
  4. 后续请求携带 Bearer Token
  5. Gateway验证JWT签名+有效期
```

### 3.2 JWT Payload设计

```json
{
  "sub": "user_uuid",
  "iss": "solra-auth",
  "iat": 1719744000,
  "exp": 1719744900,
  "roles": ["ROLE_USER"],
  "permissions": ["space:enter", "avatar:create", "chat:send"],
  "device_id": "device_hash"
}
```

### 3.3 RBAC角色设计

| 角色 | 权限范围 |
|------|----------|
| ROLE_USER | 基础功能（浏览/进入空间/创建虚拟人/聊天） |
| ROLE_CREATOR | USER权限 + 创建空间/上传资源/发布模板 |
| ROLE_PREMIUM | USER权限 + 高级虚拟人/专属空间/去广告 |
| ROLE_MODERATOR | 内容审核/举报处理/用户管理 |
| ROLE_ADMIN | 全权限（系统管理/配置/用户管理） |

### 3.4 权限矩阵

| 功能 | USER | CREATOR | PREMIUM | MODERATOR | ADMIN |
|------|------|---------|---------|-----------|-------|
| 浏览空间 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 进入空间 | ✅ | ✅ | ✅ | ✅ | ✅ |
| 创建虚拟人 | ✅(1个) | ✅(5个) | ✅(10个) | ✅ | ✅ |
| AI对话 | ✅(100次/天) | ✅(500次/天) | ✅(无限) | ✅ | ✅ |
| 创建空间 | ❌ | ✅ | ❌ | ❌ | ✅ |
| 上传资源 | ❌ | ✅ | ❌ | ❌ | ✅ |
| 审核内容 | ❌ | ❌ | ❌ | ✅ | ✅ |

## 4. 传输安全

### 4.1 TLS配置

| 配置项 | 值 |
|--------|-----|
| 协议版本 | TLS 1.3 (最低TLS 1.2) |
| 加密套件 | ECDHE-RSA-AES256-GCM-SHA384 |
| 证书 | Let's Encrypt / 商业SSL |
| HSTS | max-age=31536000; includeSubDomains |
| 证书管理 | cert-manager (K8s) 自动续期 |

### 4.2 服务间通信

| 通信方式 | 安全措施 |
|----------|----------|
| gRPC | mTLS (Istio自动管理) |
| REST | JWT + TLS |
| Kafka | SASL_SSL + ACL |
| Redis | TLS + AUTH密码 |

## 5. 数据安全

### 5.1 加密存储

| 数据类型 | 加密方式 | 密钥管理 |
|----------|----------|----------|
| 用户密码 | bcrypt (cost=12) | - |
| 手机号 | AES-256-GCM | HashiCorp Vault |
| 邮箱 | AES-256-GCM | HashiCorp Vault |
| 身份证 | AES-256-GCM + 脱敏展示 | HashiCorp Vault |
| 支付信息 | PCI DSS合规，不存储CVV | 支付网关托管 |
| 聊天记录 | AES-256-GCM (可选端到端) | Vault |

### 5.2 数据脱敏

| 场景 | 脱敏规则 |
|------|----------|
| 手机号 | 138****1234 |
| 邮箱 | a***@example.com |
| 身份证 | 320***********1234 |
| 真实姓名 | 张** |

### 5.3 数据分级

| 级别 | 定义 | 示例 | 保护措施 |
|------|------|------|----------|
| L3 绝密 | 泄露造成严重损害 | 支付密钥/根证书 | Vault + 双人控制 |
| L2 机密 | 泄露造成较大损害 | 用户密码/PII/聊天记录 | 加密存储 + 访问控制 |
| L1 内部 | 泄露造成一般损害 | 业务数据/配置 | 访问控制 + 审计日志 |
| L0 公开 | 可公开 | 空间公开信息/用户昵称 | 无特殊保护 |

## 6. 应用安全

### 6.1 输入验证

| 威胁 | 防护措施 |
|------|----------|
| SQL注入 | 参数化查询 (JPA PreparedStatement) |
| XSS | 输出编码 + CSP Header |
| CSRF | SameSite Cookie + CSRF Token |
| 文件上传 | 类型白名单 + 大小限制 + 病毒扫描 |
| SSRF | URL白名单 + 内网IP过滤 |
| 命令注入 | 禁止用户输入拼接到命令 |

### 6.2 API安全

| 措施 | 实现 |
|------|------|
| 认证 | JWT Bearer Token |
| 限流 | 用户级 + IP级 |
| 参数校验 | Bean Validation (JSR 380) |
| 响应过滤 | 敏感字段 @JsonIgnore |
| API版本 | URL路径版本 (/api/v1/) |
| 文档安全 | Swagger生产环境禁用 |

## 7. 安全运维

### 7.1 密钥管理 (HashiCorp Vault)

| 密钥类型 | 轮换周期 |
|----------|----------|
| JWT签名密钥 | 90天 |
| 数据库密码 | 30天 |
| API密钥 | 90天 |
| TLS证书 | 90天（自动续期） |
| 加密密钥 | 180天 |

### 7.2 安全监控

| 监控项 | 工具 | 告警条件 |
|--------|------|----------|
| 异常登录 | 自研 | 异地/新设备登录 |
| 暴力破解 | Fail2ban | 5次失败/分钟 |
| SQL注入 | WAF | 规则匹配 |
| 敏感数据泄露 | DLP扫描 | 正则匹配 |
| 异常流量 | Prometheus | QPS突增300% |

### 7.3 漏洞管理

| 阶段 | 措施 | 频率 |
|------|------|------|
| 开发 | IDE插件 (SonarLint) | 实时 |
| 提交 | Git Hooks (pre-commit) | 每次提交 |
| CI | SAST (SonarQube) | 每次构建 |
| 镜像 | 容器镜像扫描 (Trivy) | 每次构建 |
| 运行 | DAST (OWASP ZAP) | 每周 |
| 依赖 | SCA (Dependabot) | 每日 |
| 渗透 | 人工渗透测试 | 每季度 |

### 7.4 安全响应流程

```
发现安全事件 → 安全响应SOP
  1. 确认与定级 (15分钟内)
  2. 遏制 (立即)
  3. 根除 (24小时内)
  4. 恢复 (根据影响)
  5. 复盘与改进 (1周内)
  6. 通报 (按法规要求)
```
