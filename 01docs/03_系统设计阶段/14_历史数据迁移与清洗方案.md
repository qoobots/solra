## 2. 开发与测试数据管理

### 2.1 测试数据生成

使用 `db-faker` 工具生成符合业务规则的测试数据。

```python
# contracts/database/seeds/generate_test_data.py
from faker import Faker
import uuid, json, random

fake = Faker('zh_CN')

def generate_users(count=1000):
    users = []
    for i in range(count):
        users.append({
            'id': str(uuid.uuid4()),
            'nickname': fake.user_name(),
            'email': fake.email(),
            'status': random.choice(['ACTIVE']*8 + ['INACTIVE', 'BANNED']),
            'created_at': fake.date_time_between('-1y', 'now').isoformat()
        })
    return users
```

### 2.2 数据脱敏

开发/测试环境严禁使用生产数据。如需导入生产数据，必须脱敏：

| 字段 | 脱敏方式 | 示例 |
|------|----------|------|
| 手机号 | 随机替换 | 138****1234 → 13800000001 |
| 邮箱 | 域名替换 | user@real.com → user@test.local |
| 密码 | 统一重置 | 所有用户密码 → Test@123456 |
| 身份证 | 随机生成 | 真实号码 → 合法校验码 |
| 支付信息 | 全部清空 | - |
| 聊天记录 | 随机文本替换 | 真实对话 → Lorem Ipsum |

```sql
-- 脱敏脚本示例
UPDATE users SET
    phone = '1380000' || LPAD(row_number() OVER()::TEXT, 4, '0'),
    email = 'user' || row_number() OVER()::TEXT || '@test.local',
    password_hash = crypt('Test@123456', gen_salt('bf', 12))
WHERE id IN (SELECT id FROM users LIMIT 100);
```

## 3. 数据库版本升级迁移

### 3.1 迁移原则

| 原则 | 说明 |
|------|------|
| **向前兼容** | 新版本必须兼容旧版本数据 |
| **可回滚** | 每个迁移应有Undo脚本（生产慎用） |
| **零停机** | 使用蓝绿部署或滚动更新 |
| **先验证后上线** | Staging环境先执行迁移验证 |

### 3.2 迁移流程

```
1. 开发编写迁移脚本（V{N}__{description}.sql）
2. 提交PR → CI自动验证迁移（Dry Run）
3. 合并 → Staging自动执行
4. Staging验证通过 → 生产发布
5. 生产应用启动时Flyway自动执行
```

### 3.3 复杂迁移示例

**场景**：用户表增加 `display_id` 字段，需为所有用户生成唯一显示ID。

```sql
-- V5__add_display_id.sql
-- Step 1: 添加可空字段
ALTER TABLE users ADD COLUMN display_id VARCHAR(12);

-- Step 2: 为现有用户生成display_id
UPDATE users SET display_id = 'U' || LPAD(row_number() OVER()::TEXT, 8, '0')
WHERE display_id IS NULL;

-- Step 3: 添加非空约束和唯一索引
ALTER TABLE users ALTER COLUMN display_id SET NOT NULL;
CREATE UNIQUE INDEX idx_users_display_id ON users(display_id);
```

### 3.4 大表迁移注意事项

| 注意点 | 方案 |
|--------|------|
| 避免锁表 | 使用 `CREATE INDEX CONCURRENTLY` |
| 分批更新 | 每次更新≤10000行，间隔1秒 |
| 超时设置 | `SET lock_timeout = '5s'` |
| 监控 | 监控迁移进度和锁等待 |

```sql
-- 分批更新大表
DO $$
DECLARE
    batch_size INT := 10000;
    total_rows INT;
    offset_val INT := 0;
BEGIN
    SELECT COUNT(*) INTO total_rows FROM users WHERE display_id IS NULL;
    WHILE offset_val < total_rows LOOP
        UPDATE users SET display_id = 'U' || LPAD(nextval('display_id_seq')::TEXT, 8, '0')
        WHERE id IN (
            SELECT id FROM users WHERE display_id IS NULL
            ORDER BY created_at LIMIT batch_size
        );
        offset_val := offset_val + batch_size;
        COMMIT;
        PERFORM pg_sleep(1);
    END LOOP;
END $$;
```

## 4. 第三方数据导入

### 4.1 支持的数据源

| 数据源 | 导入工具 | 说明 |
|--------|----------|------|
| CSV/JSON | PostgreSQL COPY | 批量导入 |
| MySQL | pgloader | 异构数据库迁移 |
| MongoDB | mongoexport + 脚本 | 文档型→关系型 |
| REST API | 自定义ETL脚本 | API数据拉取 |

### 4.2 CSV批量导入

```sql
-- 准备临时表
CREATE TEMP TABLE users_import (
    phone VARCHAR(20),
    email VARCHAR(255),
    nickname VARCHAR(50)
);

-- 导入CSV
COPY users_import FROM '/data/users_import.csv' CSV HEADER;

-- 数据清洗
UPDATE users_import SET
    phone = TRIM(phone),
    email = LOWER(TRIM(email)),
    nickname = TRIM(nickname);

-- 去重
DELETE FROM users_import a USING users_import b
WHERE a.ctid > b.ctid AND a.email = b.email;

-- 导入正式表
INSERT INTO users (phone, email, nickname, password_hash, status)
SELECT
    phone,
    email,
    nickname,
    crypt('Welcome@123', gen_salt('bf', 12)),
    'ACTIVE'
FROM users_import
WHERE NOT EXISTS (
    SELECT 1 FROM users WHERE users.email = users_import.email
);
```

### 4.3 数据质量检查

导入后必须执行的数据质量检查：

| 检查项 | SQL |
|--------|-----|
| 重复数据 | `SELECT email, COUNT(*) FROM users GROUP BY email HAVING COUNT(*) > 1` |
| 空值检查 | `SELECT COUNT(*) FROM users WHERE nickname IS NULL` |
| 格式校验 | `SELECT * FROM users WHERE email !~ '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z]{2,}$'` |
| 引用完整性 | 检查外键引用是否有效 |

## 5. 数据清理

### 5.1 定期清理策略

| 数据类型 | 保留策略 | 清理方式 |
|----------|----------|----------|
| 会话记录 | 90天 | 定时任务删除 |
| 通知记录 | 180天 | 定时任务删除 |
| 验证码 | 24小时 | 定时任务删除 |
| 操作日志 | 1年 | 归档后删除 |
| 软删除数据 | 30天 | 定时任务物理删除 |

### 5.2 清理脚本

```sql
-- 清理过期会话
DELETE FROM sessions
WHERE expires_at < NOW() - INTERVAL '90 days'
   OR revoked_at < NOW() - INTERVAL '30 days';

-- 清理过期验证码
DELETE FROM verification_codes
WHERE expires_at < NOW();

-- 物理删除已注销用户数据（30天后）
DELETE FROM users
WHERE status = 'DELETED'
  AND deleted_at < NOW() - INTERVAL '30 days';
```
