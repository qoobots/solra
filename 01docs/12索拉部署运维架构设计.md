> 从代码到生产——索拉的容器化部署、Kubernetes 编排、GitOps 交付、可观测性与运维管理体系

## 一、部署运维架构总纲

### 1.1 文档定位

`07索拉应用架构设计.md` 定义了四层系统架构（L0客户端→L4基础设施），`08索拉技术架构设计.md` 给出了技术选型。本文档回答**系统怎么部署、怎么运维、怎么保证服务不劣化**。

```
运维文档层级关系：
┌─────────────────────────────────────────────────────────┐
│  07 应用架构设计    → 系统四层架构 + 部署单元定义         │
│  08 技术架构设计    → 技术栈选型 + CI/CD 流水线概要       │
│  12 部署运维架构    → K8s集群 + GitOps + 可观测性 + SLO  │  ← 本文档
└─────────────────────────────────────────────────────────┘
```

### 1.2 运维设计原则

| 原则 | 说明 | 索拉实践 |
|------|------|---------|
| **不可变基础设施** | 服务器不修改，只替换（Immutable Infrastructure） | 容器镜像即发布单元，禁止 SSH 到容器修改 |
| **声明式运维** | 期望状态用 YAML 描述，系统自动调谐 | GitOps：Git = 唯一真实源，ArgoCD 自动同步 |
| **渐进交付** | 变更渐进放量，发现问题自动回滚 | 灰度发布 5%→20%→50%→100% |
| **可观测性内建** | 日志/指标/追踪从 Day 1 构建 | OpenTelemetry 标准，所有服务统一接入 |
| **爆炸半径控制** | 故障隔离在最小范围 | 三集群（业务/AI推理/数据）独立部署 |
| **经济性** | 成本可度量、可优化 | FinOps 成本归因，按子项目/功能核算 |

### 1.3 三阶段运维策略

| 阶段 | 用户规模 | 集群规模 | 运维重点 | 自动化程度 |
|------|---------|---------|---------|-----------|
| **技术验证期（1-2年）** | < 1万 DAU | 1 集群（混合部署） | 快速迭代，人工运维可接受 | 基础 CI/CD |
| **产品验证期（3-4年）** | < 100万 DAU | 3 集群（业务/AI/数据） | 自动化覆盖率提升，SLO 建立 | GitOps + 自动扩缩容 |
| **规模验证期（5-7年）** | < 1000万 DAU | 多地域多集群 | 全自动运维，成本优化 | AIOps + 自动故障恢复 |

### 4.2 CI 流水线定义

```yaml
# .github/workflows/ci.yml (技术验证期)
name: CI Pipeline
on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Java Checkstyle
        run: ./gradlew checkstyleMain
      - name: Python Ruff
        run: ruff check .
      - name: Proto Buf Lint
        run: buf lint

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Unit Tests (Java)
        run: ./gradlew test jacocoTestReport
      - name: Unit Tests (Python)
        run: pytest --cov=. --cov-report=xml
      - name: Quality Gate
        uses: sonarsource/sonarqube-quality-gate-action@v1
        with:
          coverage: '>80%'

  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: OWASP Dependency Check
        run: ./gradlew dependencyCheckAnalyze
      - name: Trivy Image Scan
        uses: aquasecurity/trivy-action@v0.x

  build-and-push:
    needs: [lint, test, security]
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - name: Build & Push Image
        run: |
          ./gradlew jib
          cosign sign harbor.solra.io/${{ matrix.service }}:${{ github.sha }}

  deploy-staging:
    needs: build-and-push
    runs-on: ubuntu-latest
    steps:
      - name: Update Image Tag
        run: |
          cd gitops-repo
          yq -i ".images.tag = \"${{ github.sha }}\"" apps/${{ matrix.service }}/overlays/staging/kustomization.yaml
          git commit -m "chore: bump ${{ matrix.service }} to ${{ github.sha }}"
          git push
      # ArgoCD 自动同步 staging 环境
```

### 5.2 关键指标定义（RED + USE）

#### RED 指标（面向服务）

| 指标 | 定义 | 告警阈值 |
|------|------|---------|
| **Rate (请求速率)** | 每秒请求数 | 环比下降 > 50% |
| **Error (错误率)** | 5xx / 总请求数 | > 1% (P0 服务) / > 5% (P1服务) |
| **Duration (延迟)** | P50 / P95 / P99 | P99 > SLO × 1.5 |

#### USE 指标（面向资源）

| 指标 | 定义 | 告警阈值 |
|------|------|---------|
| **CPU Usage** | 使用率 | > 80% (5min) |
| **Memory Usage** | 内存使用率 | > 85% |
| **Disk Usage** | 磁盘使用率 | > 80% |
| **Network** | 网络吞吐/丢包 | 丢包率 > 0.1% |

### 5.3 监控大盘设计

| 大盘名称 | 受众 | 核心指标 |
|---------|------|---------|
| **业务概览大盘** | 全员 | DAU、空间进入量、平均停留时长、收入 |
| **核心域大盘 (AVT+SPC)** | 核心域团队 | 对话延迟/成功率、空间加载时间、推荐点击率 |
| **基础设施大盘** | SRE/DevOps | K8s 资源、数据库 QPS/延迟、GPU 利用率 |
| **安全大盘** | 安全团队 | 拦截量、攻击趋势、异常登录 |
| **成本大盘** | 管理层 | 按子项目的日/月成本、GPU 成本、存储成本 |

### 5.4 告警规则设计

| 告警名称 | 条件 | 严重级别 | 通知方式 |
|---------|------|---------|---------|
| 服务可用性下降 | 5xx率 > 1% 持续 5min | P0 Critical | 电话 + 企业微信 + PagerDuty |
| API 延迟飙升 | P99 > SLO × 2 持续 5min | P1 High | 企业微信 + PagerDuty |
| GPU 推理队列堆积 | 队列长度 > 1000 | P1 High | 企业微信 |
| 数据库连接池耗尽 | 活跃连接 > 80% | P1 High | 企业微信 |
| CPU/内存 > 90% | 持续 5min | P2 Medium | 企业微信 |
| 磁盘使用率 > 85% | 持续 | P2 Medium | 企业微信 |
| 证书即将过期 | < 15天 | P2 Medium | 企业微信 |

## 六、SLO 与错误预算

### 6.1 服务等级目标 (SLO)

| 服务 | SLI | SLO | 错误预算 (30天) |
|------|-----|-----|----------------|
| API 网关 | 可用性 (200/非5xx) | 99.9% | 43.2分钟 |
| 虚拟人对话 (AVT) | P99 延迟 < 3秒 | 99.5% | 3.6小时 |
| 空间流式加载 (SPC) | P99 加载 < 5秒 | 99.5% | 3.6小时 |
| 推荐流 (SPC) | P99 响应 < 500ms | 99.9% | 43.2分钟 |
| 状态同步 (SOC) | P99 延迟 < 200ms | 99.5% | 3.6小时 |
| 内容安全过滤 (SAF) | 拦截漏过率 | < 0.01% | — |
| 支付 (MON) | 成功率 | 99.99% | 4.3分钟 |
| 认证 (AUTH) | 登录成功率 | 99.95% | 21.6分钟 |
| 推送通知 (NOT) | 到达率 | 99% | 7.2小时 |

### 6.2 错误预算策略

```
错误预算消耗规则：
├── 预算 > 50%: 正常运维，可以接受风险性发布
├── 预算 20%-50%: 暂停非紧急发布，聚焦稳定性
├── 预算 < 20%: 冻结所有发布，全力修复可靠性
├── 预算耗尽: 启动事后复盘，制定改进计划
└── 自然恢复: 每30天重置错误预算
```

## 七、弹性与容灾

### 7.1 高可用策略

| 组件 | 高可用方案 | RPO | RTO |
|------|-----------|-----|-----|
| API 网关 | 多实例 + NLB 负载均衡 | 0 | < 1分钟 |
| 业务微服务 | 多副本 (HPA ≥ 3) + 跨 AZ | 0 | < 1分钟 |
| AI 推理服务 | GPU 集群自动扩缩容 + 降级到端侧推理 | 0 | < 2分钟 |
| MySQL | InnoDB Cluster (3节点, 1主2从) | 0 | < 30秒 (自动故障转移) |
| Redis | Redis Cluster (6节点, 3主3从) | 0 | < 30秒 |
| 对象存储 | MinIO 多站点复制 / S3 跨区域 | 0 | 自动 |
| 消息队列 | Pulsar 多 Broker + BookKeeper 多副本 | 0 | < 1分钟 |

### 7.2 降级策略

| 场景 | 降级策略 |
|------|---------|
| 云端 LLM 不可用 | 自动切端侧 SLM（功能降级，体验保持） |
| 推荐服务不可用 | 返回热门空间缓存 + 随机推荐 |
| 社交同步服务不可用 | 降级为单人空间模式 |
| 支付服务不可用 | 展示维护公告，保留订单草稿 |
| CDN 节点故障 | 自动切换备用节点 + P2P 补位 |

### 7.3 灾备方案

| 灾备级别 | 场景 | 方案 | RPO | RTO |
|---------|------|------|-----|-----|
| 同城灾备 | AZ 故障 | K8s 跨 AZ 部署，自动切换 | 0 | < 5分钟 |
| 异地灾备 | Region 故障 | 数据库异步复制，DNS 切换 | < 1分钟 | < 30分钟 |
| 数据备份 | 逻辑错误/误删 | 每日全量备份 + 增量 binlog | 可恢复到任意时间点 | < 2小时 |

### 8.1 成本归因模型

```
成本维度：
├── 计算成本
│   ├── K8s Node (按 Namespace 归因到子项目)
│   ├── GPU 推理 (按请求量归因到 AVT/SAF)
│   └── Serverless (按调用次数)
├── 存储成本
│   ├── 对象存储 (按 Bucket/子项目)
│   ├── 数据库 (按实例)
│   └── 备份存储
├── 网络成本
│   ├── CDN 流量 (按空间/子项目)
│   ├── 跨 AZ/Region 流量
│   └── NAT 网关
└── 第三方服务
    ├── 推送通道 (按消息量)
    ├── 地图 API
    └── 云服务订阅
```

### 8.2 成本优化策略

| 策略 | 说明 | 预计节省 |
|------|------|---------|
| Spot 实例 | AI 推理/批处理任务使用竞价实例 | 60-70% |
| 自动扩缩容 | 根据负载自动伸缩，避免闲置 | 30-40% |
| 存储分层 | 冷数据自动迁移至低频存储 | 50-60% |
| CDN 优化 | 空间资产边缘缓存 + P2P 分发 | 30-50% |
| Reserved Instance | 稳定负载购买预留实例 | 30-40% |
