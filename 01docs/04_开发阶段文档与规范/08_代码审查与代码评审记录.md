## 1. 代码审查流程

```
开发者提交PR → 自动CI检查 → 人工Code Review → 修改 → 批准 → 合并
```

### 1.1 审查角色

| 角色 | 职责 |
|------|------|
| 作者 | 提交PR，响应评审意见，修改代码 |
| 评审者 | 审查代码质量、逻辑正确性、安全性 |
| 模块负责人 | 核心模块必须审批 |

### 1.2 审查要求

| 要求 | 说明 |
|------|------|
| 至少1人审批 | 核心模块（auth/security/payment）至少2人 |
| CI全绿 | 编译/测试/静态分析全部通过 |
| 无未解决的讨论 | 所有评审意见需解决或回复 |
| 及时响应 | 评审意见24小时内响应 |

## 2. 审查检查清单

### 2.1 设计层面

- [ ] 是否遵循SOLID原则
- [ ] 是否有循环依赖
- [ ] 是否有过度设计
- [ ] 是否影响现有架构

### 2.2 功能层面

- [ ] 是否正确实现了需求
- [ ] 边界条件是否覆盖
- [ ] 错误处理是否完善
- [ ] 是否向后兼容

### 2.3 代码质量

- [ ] 命名是否清晰
- [ ] 代码是否简洁可读
- [ ] 是否有重复代码
- [ ] 注释是否充分且有意义

### 2.4 安全层面

- [ ] 是否有SQL注入风险
- [ ] 是否有XSS风险
- [ ] 敏感信息是否暴露
- [ ] 权限校验是否完善
- [ ] 输入是否充分校验

### 2.5 性能层面

- [ ] 是否有N+1查询
- [ ] 是否合理使用缓存
- [ ] 大循环是否有性能问题
- [ ] 数据库查询是否优化

### 2.6 测试层面

- [ ] 是否有单元测试
- [ ] 测试覆盖率是否达标(≥80%)
- [ ] 关键路径是否有测试
- [ ] 异常路径是否有测试

## 3. 审查意见分级

| 级别 | 标记 | 说明 |
|------|------|------|
| 🔴 Blocking | `[blocking]` | 必须修改才能合并 |
| 🟡 Suggestion | `[suggestion]` | 建议修改，不阻塞合并 |
| 🟢 Praise | `[praise]` | 赞赏/认可 |
| 🔵 Question | `[question]` | 提问/讨论 |

### 3.1 评审评论示例

```markdown
[blocking] 这里存在SQL注入风险，请使用参数化查询。

[suggestion] 建议将此方法拆分为两个，职责更清晰。

[praise] 这个异常处理写得很优雅！

[question] 为什么选择用Redis而不是本地缓存？有什么考虑？
```
