## 一、数据架构总纲

### 1.1 文档定位

本文档是 `07索拉应用架构设计.md` 中数据层的深化，回答"数据怎么存、怎么流、怎么管"。

```
文档层级关系：
┌─────────────────────────────────────────────────────────┐
│  07 索拉应用架构    → DDD 限界上下文 + 聚合根定义          │
│  08 索拉技术架构    → 存储技术选型 + 缓存基础设施          │
│  10 索拉数据架构    → 库表 Schema + ER 模型 + 数据流      │  ← 本文档
│  09 索拉交互设计    → 交互流程 + 状态机                   │
└─────────────────────────────────────────────────────────┘
```

### 1.2 数据架构原则

| 原则 | 说明 | 设计体现 |
|------|------|---------|
| **每子项目独立数据库** | 每个 DDD 限界上下文拥有独立 MySQL Schema，杜绝跨库 JOIN | MySQL 按子项目分库 |
| **聚合根为持久化边界** | 一次事务只修改一个聚合根，跨聚合通过领域事件最终一致 | 所有表设计以聚合根为粒度 |
| **CQRS 读写分离** | 写路径保证 ACID，读路径按需投影到缓存/搜索/分析 | 命令端 MySQL + 查询端 Redis/ES/向量DB |
| **冷热分层** | 热数据（T+0~7）在 Redis/ES，温数据（T+7~30）在 MySQL，冷数据（T+30+）归档对象存储 | 三温存储策略 |
| **隐私设计内建** | 个人数据可删改、可导出、可匿名化 | 所有用户关联字段支持软删除和匿名化 |
| **不可变日志** | 关键业务操作（支付、审核、创作发布）保留不可变审计日志 | 事件溯源 + append-only 日志表 |

#### 2.1.2 核心表 Schema

**avatars（虚拟人主表）**

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `avatar_id` | VARCHAR(64) | PK | 虚拟人唯一标识，格式 `ava_{ulid}` |
| `owner_id` | BIGINT UNSIGNED | INDEX, 可空 | 所属用户ID（系统虚拟人为NULL） |
| `name` | VARCHAR(100) | NOT NULL, UNIQUE(owner_id) | 虚拟人内部名 |
| `display_name` | VARCHAR(100) | NOT NULL | 展示名（用户可见） |
| `avatar_type` | ENUM('SYSTEM','USER','GUEST','TEMPLATE') | NOT NULL | 虚拟人类型 |
| `status` | ENUM('ACTIVE','TRAINING','PAUSED','ARCHIVED') | NOT NULL, DEFAULT 'ACTIVE' | 状态 |
| `model_version` | VARCHAR(32) | NOT NULL | 当前使用的模型版本 |
| `appearance_config` | JSON | NOT NULL | 外观配置（身体/服装/表情基线） |
| `is_public` | TINYINT(1) | NOT NULL, DEFAULT 0 | 是否公开（可被其他空间引用） |
| `created_at` | DATETIME(3) | NOT NULL | 创建时间 |
| `updated_at` | DATETIME(3) | NOT NULL | 更新时间 |

**索引**：`idx_owner_id (owner_id)`, `idx_status_type (status, avatar_type)`

**conversations（对话会话）**

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `conversation_id` | CHAR(36) | PK | UUID v7（时间排序） |
| `user_id` | BIGINT UNSIGNED | NOT NULL, INDEX | 用户ID |
| `avatar_id` | VARCHAR(64) | NOT NULL, INDEX | 虚拟人ID |
| `space_id` | VARCHAR(64) | NOT NULL | 对话发生的空间ID |
| `status` | ENUM('ACTIVE','PAUSED','ENDED') | NOT NULL, DEFAULT 'ACTIVE' | 会话状态 |

| `ended_at` | DATETIME(3) | 可空 | 结束时间 |
| `total_turns` | INT UNSIGNED | NOT NULL, DEFAULT 0 | 总轮次 |

#### 2.1.3 Redis 缓存设计 (solra_avt)

| Key Pattern | 类型 | TTL | 说明 |
|-------------|------|-----|------|
| `avt:ctx:{conv_id}` | Hash | 30min (滑动) | 对话上下文缓存（最近N轮摘要+情感状态） |
| `avt:emotion:{avatar_id}:{user_id}` | Hash | 1h | 虚拟人当前情感向量热缓存 |
| `avt:cooldown:{avatar_id}:{user_id}` | String | 动态 | 惊喜事件触发冷却（基于initiative_level计算） |
| `avt:greet:{user_id}:{space_id}` | String | 24h | 空间进入已打招呼标记（防重复） |
| `avt:memcache:{avatar_id}:{user_id}` | List (LRU) | 无 | 向量检索结果缓存（最近K条记忆） |

#### 2.1.4 向量存储设计 (Milvus)

**Collection: `avatar_memories`**

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | VARCHAR(64) | 与 MySQL memory_id 对应 |
| `embedding` | FLOAT_VECTOR(1024) | 1024维嵌入向量（BGE-M3） |
| `avatar_id` | VARCHAR(64) | 分区键 |
| `importance` | FLOAT | 用于检索时加权 |
| `memory_type` | VARCHAR(20) | 过滤条件 |
| `created_at` | INT64 | Unix 时间戳 |

**索引参数**：IVF_FLAT, nlist=1024（<100万向量）→ HNSW（>100万向量）
**分区策略**：按 `avatar_id` HASH 256分区
**检索参数**：Top-K=20, nprobe=16

## 六、数据一致性策略

### 6.1 一致性模型矩阵

| 数据操作 | 一致性要求 | 实现方式 | 最终一致窗口 |
|---------|-----------|---------|------------|
| 用户注册 | 强一致 | MySQL 单库事务 | 实时 |
| 订单支付 | 强一致 | MySQL ACID + 支付网关幂等 | 实时 |
| 空间发布 | 读后写一致 | MySQL 事务 → 异步通知 SPC/SAF | <5秒 |
| 进入空间→虚拟人打招呼 | 最终一致 | 领域事件 user.entered.space | <1秒 |
| 虚拟人记忆写入 | 最终一致 | MySQL + Milvus 双写（先MySQL后Milvus） | <5秒 |
| 多人状态同步 | 弱一致 (Redis实时) | Redis Hash + WebRTC数据通道 | <200ms |
| 排行榜更新 | 最终一致 | 定时计算 (每5分钟) | <5分钟 |
| 推荐流更新 | 最终一致 | 离线计算 + 实时结合 | <5分钟(离线) / 实时(在线) |
| 创作者结算 | 强一致 | MySQL 事务 + 对账系统 | 实时(写入) / T+1(对账) |

### 6.2 CQRS 写模型设计原则

```
写操作 → MySQL (ACID 事务)
     │
     ├──▶ 事务提交后 → 发布领域事件 (Pulsar/Kafka)
     │         │
     │         ├──▶ 事件消费者更新读模型
     │         │    · Redis 缓存更新
     │         │    · ES 索引更新
     │         │    · Milvus 向量写入
     │         │    · ClickHouse 分析写入
     │         │
     │         └──▶ 读模型最终一致窗口内可容忍短暂不一致
     │              关键业务提供"强制刷新"接口兜底
     │
     └──▶ 查询操作优先走读模型
          · 缓存命中 → 直接返回
          · 缓存miss → 查询 MySQL 从库 → 回填缓存
```

### 6.3 跨域 Saga 事务

对于需要跨子项目协调的长事务（如：空间发布 = 创作域保存 + 安全域审核 + 消费域索引），采用 Saga 编排模式：

```
┌──────────────────────────────────────────────────────────────┐
│              SpacePublishingSaga                              │
│                                                               │
│  Step 1: CRT 更新项目状态 → PUBLISHING                        │
│     │                                                         │
│     ▼                                                         │
│  Step 2: 资产打包与上传到 CDN                                  │
│     │ 失败 → Step 1 补偿: 恢复 DRAFT 状态                     │
│     ▼                                                         │
│  Step 3: SAF 自动审核 (异步)                                   │
│     │ 审核中 → 等待 (最多30分钟)                               │
│     │ 失败 → Step 2 补偿: 标记审核失败，通知作者               │
│     ▼                                                         │
│  Step 4: SPC 同步元数据快照 + 加入推荐索引                     │
│     │ 失败 → Step 3 补偿: 撤回审核                             │
│     ▼                                                         │
│  Step 5: CRT 更新状态 → PUBLISHED + 发送粉丝通知              │
│     │ 失败 → Step 4 补偿: 从推荐索引移除                       │
│     ▼                                                         │
│  完成                                                         │
└──────────────────────────────────────────────────────────────┘
```

| `solra_spc.recommendation_logs` | 按天分区 | `shown_at` | — | — | MySQL 原生分区 |
| `solra_soc.chat_messages` | 按天分区 | `created_at` | — | — | MySQL 原生分区 |
| Milvus: avatar_memories | HASH 分区 | `avatar_id` | 256 | 1024 | Milvus 原生分区 |
| Milvus: space_features | 无分区 | — | 1 | — | 规模<1亿向量无需分区 |

### 7.2 索引策略

**索引设计原则**：
- 每个查询必须有覆盖索引，Explain 分析驱动
- 复合索引列顺序：等值查询列 → 范围查询列 → 排序/覆盖列
- 避免在 JSON 列上创建索引（除非使用虚拟列）
- 分区表的索引建在分区内（LOCAL INDEX）

**高优先级索引清单**：

| 表 | 索引名 | 列 | 用途 |
|----|--------|----|------|
| `conversations` | `idx_user_avatar` | `(user_id, avatar_id)` | 用户与特定虚拟人的对话历史查询 |
| `dialogue_turns` | `idx_conv_seq` | `(conversation_id, sequence_num)` | 对话上下文拉取 |
| `space_visits` | `idx_user_entered` | `(user_id, entered_at DESC)` | 用户访问历史 |
| `space_visits` | `idx_space_entered` | `(space_id, entered_at DESC)` | 空间访问统计 |
| `recommendation_logs` | `idx_user_shown` | `(user_id, shown_at DESC)` | 用户推荐曝光去重 |
| `avatar_memories` | `idx_avatar_user` | `(avatar_id, user_id, is_deleted)` | 虚拟人-用户记忆查询 |
| `friendships` | `idx_user_a_status` | `(user_id_a, status)` | 用户A的好友列表 |
| `orders` | `idx_user_created` | `(user_id, created_at DESC)` | 用户订单历史 |
| `notifications` | `idx_user_read` | `(user_id, is_read, created_at DESC)` | 未读通知查询 |

## 八、数据安全与合规

### 8.1 数据分类分级

| 级别 | 数据示例 | 存储加密 | 传输加密 | 脱敏展示 | 可导出 | 可删除 |
|------|---------|---------|---------|---------|--------|--------|
| **公开** | 空间标题/描述/标签/发布者昵称 | 无需 | TLS | 否 | 是 | 否 |
| **内部** | 访问统计/排行榜/推荐评分 | AES-256-CBC | TLS | 否 | 是(作者) | 否 |
| **敏感** | 手机号/身份证号/真实姓名/支付信息 | AES-256-GCM (字段级) | TLS | 是(138****1234) | 受限 | 是 |
| **机密** | 密码哈希/JWT密钥/API密钥 | KMS托管 | TLS | N/A | 否 | 仅KMS操作 |

### 8.2 数据脱敏规则

| 原始数据 | 脱敏规则 | 示例 |
|---------|---------|------|
| 手机号 | 保留前3后4 | `138****1234` |
| 身份证号 | 保留前6后4 | `110101********1234` |
| 真实姓名 | 保留姓，名用* | `张**` |
| IP地址 | 保留前3段 | `192.168.1.*` |
| 支付账号 | 保留后4位 | `****5678` |

### 8.3 用户数据权利实现

| 用户权利 | 实现方式 | 响应时效 |
|---------|---------|---------|
| **查阅权** | `/api/v1/user/data/export` - 异步导出压缩包 | <48小时 |
| **更正权** | 各业务API支持修改个人基础信息 | 实时 |
| **删除权** | 软删除→30天宽限→匿名化处理→不可逆删除 | 30天 |
| **携带权** | 结构化JSON/CSV格式导出 | <48小时 |
| **限制处理** | 账户冻结（保留数据但停止所有自动化处理） | 实时 |

### 8.4 审计日志设计

```sql
CREATE TABLE solra_audit.audit_logs (
    audit_id        CHAR(36)        PRIMARY KEY,
    trace_id        CHAR(32)        NOT NULL,       -- 关联全链路追踪
    user_id         BIGINT UNSIGNED,                 -- 操作者 (可空=系统)
    action          VARCHAR(128)    NOT NULL,        -- 操作类型
    resource_type   VARCHAR(64)     NOT NULL,        -- 资源类型
    resource_id     VARCHAR(64),                     -- 资源ID
    old_value_json  JSON,                            -- 变更前快照
    new_value_json  JSON,                            -- 变更后快照
    ip_address      VARCHAR(45),                     -- 操作者IP
    user_agent      VARCHAR(512),                    -- User-Agent
    result          ENUM('SUCCESS','FAILURE','DENIED') NOT NULL,
    created_at      DATETIME(3)     NOT NULL,
    
    INDEX idx_user_action (user_id, created_at),
    INDEX idx_resource (resource_type, resource_id),
    INDEX idx_action_time (action, created_at)
) ENGINE=InnoDB 
  PARTITION BY RANGE (TO_DAYS(created_at)) (...)
  COMMENT='不可变审计日志——仅追加，禁止UPDATE/DELETE';
```

审计日志覆盖操作：用户注册/注销、登录/登出、角色变更、空间创建/发布/删除、虚拟人训练/删除、订单创建/退款、用户举报/申诉、管理员操作。

## 九、数据质量与监控

### 9.1 数据质量检查项

| 检查项 | 规则 | 检测频率 | 处理方式 |
|--------|------|---------|---------|
| 孤儿记录 | 无对应外键的孤儿子记录 | 每小时 | 告警+自动软删除 |
| 重复记录 | 唯一约束外的业务重复（如同用户同时两个ACTIVE订阅） | 每小时 | 告警+人工处理 |
| 数据完整性 | 必要字段NULL值检查 | 实时(写入时) | 拒绝写入 |
| 数值范围 | 情感值[0,1]、评分[0,5]、金额>0 | 实时(写入时) | 拒绝写入 |
| 数据延迟 | MySQL→从库复制延迟 | 每分钟 | >10秒告警 |
| 存储容量 | 磁盘使用率 | 每5分钟 | >80%告警, >90%紧急扩容 |
| 慢查询 | >200ms的SQL | 实时 | 记录+每周Review |
