# 索拉（Solra）应用架构设计文档

> 基于领域驱动设计（DDD）的分布式应用架构——从战略设计到战术落地的完整技术蓝图

---

## 一、架构设计总纲

### 1.1 架构愿景

索拉的架构设计服务于一个核心理念：**构建一个可感知存在、可自由进入、可深度交互的3D空间社交网络**。技术架构必须同时满足三个看似矛盾的目标：

| 目标 | 技术挑战 | 架构策略 |
|------|---------|---------|
| **低延迟进入**（<2秒加载200MB空间） | 移动端网络波动 + 3D资产体积大 | 流式加载管线 + 预测性预加载 + CDN边缘分发 |
| **高保真AI交互**（虚拟人实时对话） | GPU推理成本高 + 端侧算力有限 | 端侧SLM + 云端LLM混合推理架构 |
| **规模化服务**（千万DAU可用性不劣化） | 状态同步 + 空间分发 + 推理并发 | 事件驱动 + 无状态服务 + 边缘计算 |

### 1.2 架构设计原则

| 原则 | 说明 | DDD映射 |
|------|------|---------|
| **子项目独立自治** | 每个子项目（限界上下文）拥有独立的模型、数据库、部署单元 | 每个子项目=独立微服务群 |
| **事件驱动松耦合** | 子项目之间通过领域事件异步通信，避免同步耦合 | 领域事件发布/订阅 |
| **端边云协同** | 推理在端侧、计算在边缘、数据在云端 | CQRS读写分离 |
| **流式优先** | 空间加载、对话推理、多人同步均采用流式设计 | 响应式架构 |
| **安全内建** | 内容安全、数据安全从架构层内置，非事后叠加 | 防腐层（ACL）模式 |
| **可观测性一等公民** | 全链路追踪、指标、日志三位一体，从Day 1构建 | 基础设施子项目支撑 |

### 1.3 架构风格选择

索拉采用 **微服务 + 事件驱动 + CQRS** 的混合架构风格：

```

┌─────────────────────────────────────────────────────────────┐
│                                                              │
│   ┌─────────────────┐     ┌─────────────────────────────┐   │

│   │   (按子项目      │     │   (跨子项目异步通信)         │   │
│   │    拆分服务)     │     │                             │   │
│   └────────┬────────┘     └────────────┬────────────────┘   │
│            │                           │                      │
│            └───────────┬───────────────┘                      │
│                        │                                      │
│                        ▼                                      │
│            ┌───────────────────────┐                         │
│            │   CQRS (命令查询分离)  │                         │
│            │   读写分离 + 事件溯源   │                         │
│            └───────────────────────┘                         │
│                                                              │
│   选型理由：                                                   │
│   · 微服务：10个子项目的独立演进                                │
│   · 事件驱动：空间进入/退出/互动事件天然适合事件总线                │
│   · CQRS：空间读取（高频）与空间编辑（低频）天然读写分离             │
└─────────────────────────────────────────────────────────────┘
```

#### 3.2.2 空间流式加载管线

```
空间文件结构（~200MB 基准）
┌────────────────────────────────────┐
│  space_manifest.json    (~1KB)     │ ← 元数据：空间ID/版本/作者/依赖
│  scene_structure.bin    (~10MB)    │ ← 场景结构：网格/材质/LOD层级
│  textures_pack.ktx      (~80MB)    │ ← 纹理包：KTX2压缩，多级mipmap
│  models/                           │
│    ├─ furniture.bin      (~40MB)   │ ← 家具模型
│    ├─ decorations.bin    (~30MB)   │ ← 装饰物
│    └─ avatars.bin        (~20MB)   │ ← 空间内虚拟人模型
│  audio/                            │
│    ├─ ambient.ogg        (~8MB)    │ ← 环境音
│    └─ bgm.ogg            (~10MB)   │ ← 背景音乐
│  lighting.json           (~1KB)    │ ← 灯光配置
└────────────────────────────────────┘

流式加载时序：
t=0ms    ─── 点击进入空间
t=0-50ms ─── 下载manifest → 解析依赖 → 确定加载顺序
t=50-200ms ─ 下载scene_structure → GPU端构建场景骨架（白模渲染）
t=200-500ms ─ 流式加载textures_pack（低mipmap优先）→ 贴图出现
t=500-1000ms ─ 加载furniture + decorations → 物体依次出现
t=1000-1500ms ─ 加载avatars模型 → 虚拟人可见
t=1500-2000ms ─ 加载audio资源（后台） → BGM淡入
              → 用户已可交互 ✓

关键优化：
· 分块压缩：每个资源独立压缩，支持乱序+断点续传
· 渐进纹理：先加载64×64低清，再逐步替换为高清
· LOD自动：根据设备GPU能力动态选择模型精度
· 预测预加载：退出当前空间时，后台预加载下一个推荐空间
· P2P加速：热门空间的资源可通过P2P网络加速分发
```

## 十、架构决策记录（ADR）

### ADR-001：为什么选择自研流式加载框架而不是复用现有方案

| 属性 | 值 |
|------|-----|
| **状态** | 已采纳 |
| **背景** | 空间文件200-900MB，需在移动端<2秒内进入可交互状态。现有的glTF/USDZ加载器不支持增量流式加载，Unity/UE的AssetBundle设计目标不是移动端网络流式场景。 |
| **决策** | 自研流式加载管线，设计专有分块格式和渐进加载协议。 |
| **后果** | + 可精确控制加载优先级和带宽利用；+ 可实现预测性预加载；- 需要长期维护专有格式和对应的创作工具导出插件。 |

### ADR-002：为什么选择端侧+云端混合推理而不是纯云端

| 属性 | 值 |
|------|-----|
| **状态** | 已采纳 |
| **背景** | 虚拟人实时对话要求<1秒延迟，纯云端推理受网络延迟影响波动大，且大规模并发的GPU成本不可控。 |
| **决策** | 端侧部署1-3B SLM处理日常对话，云端7-13B LLM处理复杂推理。推理调度层按对话复杂度动态路由。 |
| **后果** | + 日常对话延迟<500ms；+ 可节省50-70%云端推理成本；- 需维护端侧模型版本与设备兼容矩阵；- 低端机型需降级为纯云端模式。 |

### ADR-003：为什么选择微服务+事件驱动而不是单体架构

| 属性 | 值 |
|------|-----|
| **状态** | 已采纳 |
| **背景** | 索拉包含10个子项目（DDD限界上下文），各子项目的迭代速度、资源需求、团队配置差异大。技术验证期可能只需部分子项目，产品验证期才全部上线。 |
| **决策** | 按DDD限界上下文拆分为独立微服务（每个子项目=独立微服务群），通过事件总线异步集成。技术验证期可只部署P0子项目。 |
| **后果** | + 各子项目独立演进、独立扩缩容；+ 按需部署降低早期成本；- 分布式系统的固有复杂性（网络延迟、数据一致性、运维复杂度）。 |

### ADR-004：为什么选择Pulsar而不是Kafka作为事件总线

| 属性 | 值 |
|------|-----|
| **状态** | 待验证（技术验证期评估） |
| **背景** | 索拉的领域事件需要多租户隔离（不同环境/团队）、分层存储（历史事件低成本归档）、低延迟+高吞吐。 |
| **决策** | 首选Apache Pulsar，备选Kafka。Pulsar的多租户、分层存储、Geo-Replication特性更匹配索拉的事件驱动架构需求。 |
| **后果** | + 多环境隔离部署更简单；+ 长期事件归档成本低；- 运维复杂度略高于Kafka；- 中文社区和人才储备小于Kafka。 |

### ADR-005：为什么空间预览卡片在服务端渲染而不是客户端渲染

| 属性 | 值 |
|------|-----|
| **状态** | 已采纳 |
| **背景** | 空间预览卡片需要在退出空间后<300ms内展示下一个空间的3D动态预览。客户端渲染需要先加载完整空间数据，无法满足延迟要求。 |
| **决策** | 服务端（CDN边缘节点）预渲染3D快照，客户端直接请求渲染好的预览图片/轻量3D模型。 |
| **后果** | + 预览卡片加载<300ms；- 需要边缘GPU渲染节点；- 预览更新频率受限（按需渲染，非实时）。 |

### ADR-006：虚拟人记忆存储选择向量数据库的理由

| 属性 | 值 |
|------|-----|
| **状态** | 已采纳 |
| **背景** | 虚拟人长期记忆需要基于语义相似度检索相关记忆注入对话上下文（RAG模式）。传统关系型数据库无法高效处理向量相似度检索。 |
| **决策** | 使用Milvus/Qdrant作为记忆向量存储，MySQL存储记忆元数据，两者通过memory_id关联。 |
| **后果** | + 毫秒级语义检索；+ 支持记忆重要度排序和自动衰减；- 增加数据一致性维护成本（双写）；- 向量数据库运维复杂度。 |

## 十一、架构风险与缓解措施

| 风险 | 影响 | 概率 | 架构层面缓解措施 |
|------|------|------|----------------|
| 端侧推理延迟不达标 | 虚拟人对话体验崩盘 | 中 | 模型路由器支持纯云端降级；技术验证期先做异步文本模式再叠加TTS |
| 3D流式加载不稳定 | 空间进入时间>5秒 | 中高 | 渐进加载策略（先白模后纹理）；离线预缓存；低端机轻量模式 |
| 多人同步弱网体验差 | 社交场景不可用 | 中 | 先验证2人同步；客户端预测+插值补偿；弱网自动降级为异步社交 |
| GPU推理成本失控 | 毛利率被严重侵蚀 | 中高 | 端侧推理优先；推理路由器的成本感知调度；FinOps实时成本监控 |
| 分布式系统运维复杂 | 故障恢复慢 | 中 | 服务网格（Istio）统一治理；GitOps自动化部署；完善的Runbook |
| 向量数据库扩展瓶颈 | 虚拟人记忆检索变慢 | 低 | 按虚拟人分区扩展；热点虚拟人独立分片；检索缓存层 |
