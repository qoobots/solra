# 索拉（Solra）安全架构设计文档

> 纵深防御 + 零信任内建——索拉的认证授权、内容安全、数据保护、合规治理的体系化安全蓝图

## 一、安全架构总纲

### 1.1 安全架构定位

`07索拉应用架构设计.md` 在 L1-L6 层概述了安全防御层次，`08索拉技术架构设计.md` 给出了加密和认证的技术实现。本文档将安全作为**一等架构关注点**，系统化定义：

- **纵深防御体系**：六层防御如何协同工作
- **零信任架构落地**：微服务间认证、服务网格安全、API 安全策略
- **内容安全闭环**：从实时过滤到人工审核的完整链路
- **数据安全与隐私合规**：分类分级、脱敏、合规矩阵
- **安全运营体系**：漏洞管理、事件响应、红蓝演练

```

```

### 1.2 安全设计原则

| 原则 | 说明 | 索拉实践 |
|------|------|---------|
| **纵深防御** | 不依赖单一防线，每一层都假设上一层已被突破 | 传输→认证→内容→数据→API→合规 六层独立 |
| **零信任内建** | 不信任任何网络边界，每次请求都需认证授权 | 微服务间 mTLS + Spiffe 身份，API 网关统一鉴权 |
| **最小权限** | 每个服务/用户仅拥有完成工作所需的最小权限 | RBAC 五角色 + 服务账号细粒度 IAM |
| **安全左移** | 安全从设计阶段开始，不是上线前的检查项 | 威胁建模→安全评审→SAST→DAST 全流水线 |
| **默认拒绝** | 未明确允许的访问一律拒绝 | API 网关默认拒绝，通过策略白名单放行 |
| **隐私设计** | 数据保护嵌入产品设计，非事后打补丁 | 端侧推理减少云端数据暴露；差分隐私用于用户行为分析 |

## 二、L1 传输安全层

### 2.1 TLS 策略

| 场景 | 最低版本 | 加密套件 | 说明 |
|------|---------|---------|------|
| 客户端→API网关 | TLS 1.3 | TLS_AES_256_GCM_SHA384 | 禁用 TLS 1.2 以下 |
| API网关→微服务 | TLS 1.3 + mTLS | 同上 | 服务网格 Istio 管理 |
| 微服务间 | mTLS 1.3 | 同上 | Spiffe 身份，自动轮换 |
| 数据库连接 | TLS 1.2+ | 服务端证书验证 | 强制 SSL |
| Redis 连接 | TLS 1.2+ | 服务端证书验证 | Redis 7 TLS |
| 消息队列 (Pulsar) | TLS 1.3 | 客户端证书认证 | Broker-Client 双向认证 |
| CDN 边缘→用户 | TLS 1.3 | ECDSA P-256 | Let's Encrypt 自动续期 |

### 2.2 证书管理

| 证书类型 | 管理方式 | 有效期 | 轮换策略 |
|---------|---------|--------|---------|
| 外部域名 | Let's Encrypt / cert-manager | 90天 | 自动续期（到期前30天） |
| 内部服务 | Istio Citadel / Spiffe | 24小时 | 自动轮换 |
| 数据库证书 | KMS 签发 | 1年 | 到期前30天告警→滚动更新 |
| 客户端证书钉扎 | App 内置公钥 Hash | 与App版本绑定 | App 更新时刷新 |

### 2.3 HSTS 与证书钉扎

| 措施 | 配置 | 说明 |
|------|------|------|
| HSTS | `max-age=31536000; includeSubDomains; preload` | 强制浏览器 HTTPS |
| 证书透明度 | CT (Certificate Transparency) 监控 | 检测恶意证书签发 |
| 移动端钉扎 | 公钥 Hash 内置 | 防止中间人攻击 |
| 钉扎后备 | 至少一个备用 Hash | 证书更新时兜底 |

### 3.2 JWT 认证详细设计

#### Token 结构

```json
// AccessToken Payload
{
  "sub": "user_1234567890",        // 用户唯一标识
  "role": "creator",               // RBAC 角色
  "did": "device_abc123",          // 设备标识
  "iat": 1719619200,               // 签发时间
  "exp": 1719620100,               // 过期时间 (15min)
  "jti": "uuid-v4-random"          // Token 唯一标识（防重放）
}

// RefreshToken Payload
{
  "sub": "user_1234567890",
  "did": "device_abc123",
  "iat": 1719619200,
  "exp": 1720224000,               // 过期时间 (7天)
  "jti": "uuid-v4-random",
  "family": "rf_family_xyz"        // Token 家族（旋转检测）
}
```

#### Token 生命周期

```
┌──────────┐     登录成功        ┌──────────────┐
│ 未认证    │ ─────────────────► │  已认证       │
└──────────┘                    │  (15min       │
                                │   AccessToken)│
                                └──────┬───────┘
                                       │ AccessToken 即将过期
                                       ▼
                                ┌──────────────┐
                                │  Token 刷新   │
                                │  (RefreshToken│
                                │   换新 AT)    │
                                └──────┬───────┘
                                       │ RefreshToken 过期/吊销
                                       ▼
                                ┌──────────────┐
                                │  需重新登录    │
                                └──────────────┘
```

#### RefreshToken 旋转检测（防泄漏）

| 场景 | 行为 |
|------|------|
| 正常使用 | 用旧 RT 换新 AT+RT，旧 RT 立即失效 |
| RT 被重复使用（旧 RT 已被使用过） | 整个 Token Family 全部失效，告警通知用户 |
| RT 过期 | 需重新登录 |

### 4.2 内容安全检测维度

| 内容类型 | 检测维度 | 技术方案 | 延迟要求 | 拦截率目标 |
|---------|---------|---------|---------|-----------|
| 虚拟人对话 | 色情/暴力/违法/政治敏感/诱导自杀 | 多级敏感词库 + BERT 文本分类 + LLM 意图检测 | < 200ms | > 99.9% |
| 3D 场景模型 | 色情模型/违禁符号/恐怖元素 | 多模态视觉模型 (ViT + CLIP) | < 5秒 | > 95% |
| 纹理/贴图 | 色情图像/暴力/违法标识 | 图像分类模型 + OCR | < 3秒 | > 98% |
| 空间文字 | 违规标语/敏感信息 | 敏感词库 + NLP | < 1秒 | > 99% |
| 用户头像/昵称 | 违规头像/昵称 | 图像+NLP 审核 | < 1秒 | > 99% |
| 音频（语音聊天） | 违规语音内容 | ASR + 文本审核 | 实时 (< 500ms) | > 95% |

### 4.3 分级处理策略

| 风险等级 | 定义 | 自动动作 | 人工介入 |
|---------|------|---------|---------|
| **L0 安全** | 确认无害内容 | 自动放行 | 无 |
| **L1 低风险** | 疑似但不确定 | 放行 + 标记（入人工抽检队列） | 抽样复审 |
| **L2 中风险** | 较可能违规 | 拦截 + 通知创作者 + 入审核队列 | 必须人工复审 |
| **L3 高风险** | 明确违规（色情/暴力/违法） | 立即拦截 + 封禁/限权 + 证据留存 | 复核确认处罚 |
| **L4 红线** | 国家安全/极端违法 | 立即拦截 + 冻结账户 + 自动上报 + 证据链保存 | 运营总监/法务介入 |

### 4.4 人工审核工作台

| 功能 | 说明 |
|------|------|
| 审核队列 | 按优先级（L3 > L2 > L1）+ 时效排序 |

## 五、L4 数据安全层

### 5.1 数据分类分级

| 级别 | 定义 | 示例数据 | 存储加密 | 传输加密 | 访问控制 | 审计 |
|------|------|---------|---------|---------|---------|------|
| **S4 核心** | 泄露导致生存危机 | 用户密码 hash、支付密钥、JWT 签名密钥 | AES-256-GCM + KMS | TLS 1.3 + mTLS | 仅安全服务 | 全量 |
| **S3 敏感** | 泄露导致严重损害 | 手机号、身份证号、实名信息、支付记录 | AES-256-GCM 字段级 | TLS 1.3 | RBAC + IP 白名单 | 全量 |
| **S2 内部** | 泄露导致轻微损害 | 用户昵称、头像、对话内容、空间数据 | 存储层加密 | TLS 1.3 | RBAC | 抽样 |
| **S1 公开** | 泄露无损害 | 公开空间信息、排行榜、用户公开展示 | 无需额外加密 | TLS | 公开 | 无 |

## 六、L5 API 安全 / 应用安全层

### 6.1 API 安全策略

| 防护措施 | 实现方式 | 配置 |
|---------|---------|------|
| 认证 | JWT 验证（APISIX jwt-auth 插件） | 所有业务 API 强制 |
| 限流 | 令牌桶（APISIX limit-req） | 用户: 100 req/s, IP: 500 req/s |
| 熔断 | 断路器（APISIX api-breaker） | 错误率 > 50% → 熔断 30s |
| IP 黑名单 | APISIX ip-restriction | 恶意 IP 自动加入 |
| 输入校验 | API 网关 + 业务层双重校验 | JSON Schema 验证 |
| CORS | APISIX cors 插件 | 仅允许 *.solra.io |
| CSRF | SameSite=Strict Cookie + CSRF Token | 状态变更接口 |
| 请求大小限制 | APISIX 限制 | 普通 API ≤ 10MB, 上传 ≤ 500MB |

### 6.2 WAF 与 DDoS 防护

| 爬虫防护 | 恶意抓取/数据爬取 | 设备指纹 + 行为分析 + 验证码 |
| API 滥用 | 撞库/扫号/恶意注册 | 风控引擎 + 设备指纹 |

## 八、零信任架构

### 8.1 零信任核心原则在索拉的落地

| 零信任原则 | 索拉落地 |
|-----------|---------|
| 永不信任，始终验证 | 每次 API 调用都验证 JWT；微服务间 mTLS + Spiffe |
| 最小权限 | 每个服务仅能访问其需要的数据库/API；每用户 RBAC 最小角色 |
| 假设已被入侵 | 网络微分段；异常行为检测；爆炸半径控制 |
| 持续验证 | 设备指纹变化 → 强制重新认证；异地登录 → 二次验证 |

### 8.2 服务间零信任

```
┌────────────────────────────────────────────────────────────┐
│                  服务间零信任架构                            │
│                                                             │
│  ┌─────────┐        ┌──────────────┐        ┌─────────┐   │
│  │ 服务 A  │──mTLS──►│  服务网格     │──mTLS──►│ 服务 B  │   │
│  │         │        │  (Istio)      │        │         │   │
│  │ Spiffe: │        │              │        │ Spiffe: │   │
│  │ spiffe://│        │ 认证 + 授权   │        │ spiffe://│  │
│  │ solra.io│        │ 流量加密      │        │ solra.io│  │
│  │ /svc/avt│        │ 策略执行      │        │ /svc/spc│  │
│  └─────────┘        └──────────────┘        └─────────┘   │
│                                                             │
│  授权策略 (Istio AuthorizationPolicy):                       │
│  · AVT 服务 → 可调用 LLM 推理服务                            │
│  · SPC 服务 → 可读取空间资产存储                             │
│  · 任何服务 → 不可直接访问其他服务的数据库                    │
│  · 外部 → 仅能通过 API 网关访问                              │
└────────────────────────────────────────────────────────────┘
```
