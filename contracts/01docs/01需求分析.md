# 协议定义层（Contracts）需求分析

> 索拉 Monorepo 协议定义子项目——定义跨子项目通信的 Proto/Event/OpenAPI 契约，是所有服务与客户端交互的"宪法"

---

## 一、需求背景

索拉采用微服务+事件驱动架构，10+个微服务、5个AI服务、3个客户端、1个C++核心引擎之间需要严格类型安全的通信契约。协议定义层（contracts/）是 Monorepo 的"单一事实来源"（Single Source of Truth）。

### 1.1 核心痛点

| 痛点 | 说明 | 后果 |
|------|------|------|
| 类型不一致 | 服务A和服务B对 User 结构的定义不同 | 运行时反序列化失败 |
| 接口漂移 | 服务接口变更未同步到调用方 | 生产事故 |
| 代码重复 | 每个服务自己维护 DTO | 维护成本高 |
| 协议演进混乱 | 缺少版本管理策略 | 不兼容变更无法检测 |

### 1.2 协议层定位

```
contracts/ 是 Monorepo 的"语言层"：
  ├── 对上：客户端通过 Proto/OpenAPI 理解服务能力
  ├── 对下：服务通过 Proto 保证接口一致性
  └── 对内：子项目通过 Event 进行异步协作
```

## 二、功能需求

### 2.1 Proto 协议定义（10个协议域）

| 编号 | 协议域 | 优先级 | 说明 |
|------|--------|--------|------|
| CT-001 | 公共模型 (common) | P0 | User/Token/SpaceId/Timestamp 等全项目共享类型 |
| CT-002 | 虚拟人交互 (avt) | P0 | Conversation/Avatar/DialogueTurn/EmotionState |
| CT-003 | 空间消费 (spc) | P0 | Space/Recommendation/PreviewCard/StreamChunk |
| CT-004 | 身份认证 (auth) | P0 | UserAccount/LoginSession/OAuthToken/Permission |
| CT-005 | 内容安全 (saf) | P0 | ReviewCase/SafetyScore/Appeal/ModerationResult |
| CT-006 | 空间创作 (crt) | P1 | SpaceProject/Asset/Template/AIToolRequest |
| CT-007 | 社交在场 (soc) | P1 | Session/Presenter/Gesture/SyncState |
| CT-008 | 用户成长 (grw) | P1 | UserProfile/Achievement/FaithLevel/ActivityLog |
| CT-009 | 通知消息 (not) | P1 | Notification/PushMessage/InboxMessage/Subscription |
| CT-010 | 商业化 (mon) | P2 | Order/Subscription/VirtualItem/Transaction |

### 2.2 事件定义

| 编号 | 事件集 | 优先级 | 说明 |
|------|--------|--------|------|
| CT-011 | 核心领域事件 | P1 | SpaceEntered/UserLeftSpace/AvatarSpoke 等跨子项目事件 |
| CT-012 | 事件版本管理 | P3 | Avro schema 兼容策略 + 事件迁移工具 |

### 2.3 OpenAPI 规范

| 编号 | API定义 | 优先级 | 说明 |
|------|---------|--------|------|
| CT-013 | 服务间 REST API | P2 | 各微服务暴露的 REST 接口 OpenAPI 3.0 规范 |
| CT-014 | 客户端 BFF API | P2 | 移动端/Web 端调用的 BFF 层接口规范 |

## 三、非功能需求

| 维度 | 要求 | 度量标准 |
|------|------|---------|
| **向后兼容** | Proto 变更不破坏已有消费者 | buf breaking 检查通过 |
| **代码生成质量** | 生成的代码直接可编译 | Java/TypeScript/Python/C++ 零错误 |
| **文档自描述** | Proto 注释即文档 | proto-doc 自动生成 HTML |
| **变更可追溯** | 每次协议变更可追踪 | Git 历史 + CHANGELOG |
| **多语言支持** | 统一 Proto 生成 Java/TS/Python/C++/Swift/Kotlin | buf generate 全语言通过 |

---

## 四、用户/调用方角色

| 角色 | 使用方式 | 关注点 |
|------|---------|--------|
| 服务开发者 | 引用 Proto 生成的代码 | 接口一致性 |
| 客户端开发者 | 通过 OpenAPI/BFF 理解接口 | 文档准确性 |
| 架构师 | 审查协议变更 PR | 架构一致性 |
| CI/CD | 自动运行 buf lint/breaking | 变更安全性 |
