# 索拉（Solra）质量保障设计文档

> 质量左移，缺陷右溯——索拉的测试金字塔、质量门禁、验收标准矩阵与质量度量的体系化设计

---

## 一、质量保障总纲

### 1.1 文档定位

索拉的质量保障不是测试团队在开发结束后才介入的"找Bug"活动，而是从需求到上线的**全链路质量内建**。本文档定义：

- **测试策略**：测试金字塔、各层测试的占比与覆盖目标
- **质量门禁**：每个阶段进入下一阶段的必须满足的质量条件
- **验收标准矩阵**：78 条功能按子项目的可度量验收标准
- **特殊场景测试**：3D渲染、AI对话、空间加载、实时同步的专项测试方案
- **质量度量**：缺陷率、覆盖率、MTTR 等持续改进指标

```
质量文档层级关系：
┌──────────────────────────────────────────────────────────┐
│  04 需求分析         → 非功能需求（性能/可靠性/安全）     │
│  06 功能清单进度      → 78 条功能验收标准摘要             │
│  13 质量保障设计      → 全链路测试策略 + 质量门禁         │  ← 本文档
└──────────────────────────────────────────────────────────┘
```

### 1.2 质量设计原则

| 原则 | 说明 | 索拉实践 |
|------|------|---------|
| **质量左移** | 越早发现问题，代价越小 | 需求评审→安全检查→单元测试→集成测试，层层左移 |
| **测试金字塔** | 底层多、上层少，金字塔而非冰淇淋筒 | 单元60% : 集成25% : E2E10% : 手工探索5% |
| **可度量** | 质量必须可度量才能改进 | 覆盖率/缺陷密度/MTTR/逃逸率等量化指标 |
| **自动化优先** | 可自动化的测试绝不手工执行 | CI 全自动：单测+集成+安全扫描+性能基线 |
| **风险驱动** | 按风险分配测试投入，核心域投入最多 | 核心域 (AVT/SPC) 测试投入占 50% |
| **验收标准即契约** | 需求文档中的验收标准 = 测试用例的直接来源 | 06 功能清单的验收标准摘要 = 验收测试基础 |

### 1.3 三阶段测试策略

| 阶段 | DAU规模 | 测试重点 | 自动化率目标 |
|------|--------|---------|------------|
| **技术验证期** (1-2年) | < 1万 | 核心管线功能正确性、手动探索性测试为主 | 60% |
| **产品验证期** (3-4年) | < 100万 | 回归自动化 + 性能测试 + 兼容性测试 | 80% |
| **规模验证期** (5-7年) | < 1000万 | 混沌工程 + 全链路压测 + AI辅助测试 | 90%+ |

### 2.2 各层测试策略详解

#### 2.2.1 单元测试（60% 占比）

| 服务 | 测试框架 | 覆盖目标 | Mock策略 | 运行时间 |
|------|---------|---------|---------|---------|
| Java 业务服务 | JUnit 5 + Mockito + AssertJ | 行覆盖率 > 80% | Mock 外部依赖（DB/API/MQ） | < 5分钟 |
| Python AI 服务 | pytest + pytest-mock | 行覆盖率 > 80% | Mock 模型加载（测试用轻量模型） | < 5分钟 |
| C++ Core SDK | Google Test (gtest) | 行覆盖率 > 70% | Mock 网络层/硬件抽象 | < 10分钟 |
| TypeScript 前端 | Vitest + Testing Library | 行覆盖率 > 70% | Mock API 调用 | < 5分钟 |

**Java 单元测试示例**：

```java
@ExtendWith(MockitoExtension.class)
class MemoryRecallServiceTest {

    @Mock
    private MemoryRepository memoryRepository;

    @Mock
    private MilvusClient milvusClient;

    @InjectMocks
    private MemoryRecallService recallService;

    @Test
    @DisplayName("当用户cue触发关键词时，应召回最近30天相关记忆")
    void shouldRecallMemories_within30Days_whenKeywordMatched() {
        // Given
        String userId = "user_123";
        String cue = "上次我们聊到海边";

        List<Memory> expectedMemories = List.of(
            new Memory("海边旅行", "2026-06-15", 0.85)
        );
        when(memoryRepository.findByUserAndTimeRange(userId, -30, 0))
            .thenReturn(expectedMemories);

        // When
        List<Memory> result = recallService.recall(userId, cue);

        // Then
        assertThat(result).hasSize(1);
        assertThat(result.get(0).getTopic()).isEqualTo("海边旅行");
        assertThat(result.get(0).getRelevanceScore()).isGreaterThan(0.8);
    }
}
```

**Python AI 服务测试示例**：

```python
@pytest.fixture
def safety_filter():
    """使用轻量级规则模型替代完整模型加速测试"""
    return DialogueSafetyFilter(model="rule-based-light")

class TestDialogueSafetyFilter:
    def test_should_block_violent_content(self, safety_filter):
        result = safety_filter.filter("我要伤害别人")
        assert result.blocked is True
        assert result.reason == "violence"

    def test_should_allow_normal_conversation(self, safety_filter):
        result = safety_filter.filter("今天天气真好")
        assert result.blocked is False

    def test_should_complete_within_latency_budget(self, safety_filter):
        start = time.time()
        safety_filter.filter("这是一段正常的对话文本 " * 50)
        elapsed = time.time() - start
        assert elapsed < 0.2  # < 200ms SLA
```

## 三、质量门禁

### 3.1 门禁矩阵

```
代码提交 → PR 创建 → 代码审查 → 合并 → 构建 → 部署
    │          │          │        │      │      │
    │    ┌─────▼─────┐    │        │      │      │
    │    │ 门禁 1     │    │        │      │      │
    │    │ 静态检查   │    │        │      │      │
    │    └───────────┘    │        │      │      │
    │              ┌──────▼──────┐ │      │      │
    │              │ 门禁 2       │ │      │      │
    │              │ 测试+覆盖率  │ │      │      │
    │              └─────────────┘ │      │      │
    │                        ┌─────▼──────┐     │
    │                        │ 门禁 3      │     │
    │                        │ 安全扫描    │     │
    │                        └────────────┘     │
    │                                     ┌─────▼──────┐
    │                                     │ 门禁 4      │
    │                                     │ 构建+镜像   │
    │                                     └────────────┘
    │                                               ┌─────▼──────┐
    │                                               │ 门禁 5      │
    │                                               │ 部署验证    │
    │                                               └────────────┘
```

| 门禁 | 阶段 | 条件 | 阻断级别 |
|------|------|------|---------|
| **门禁1: 静态检查** | PR 创建 | Lint 0 error + Checkstyle/ESLint/Ruff 通过 | 阻断合并 |
| **门禁2: 测试+覆盖率** | PR 创建 | 所有单元测试通过 + 行覆盖率 ≥ 80% + 新增代码覆盖率 ≥ 90% | 阻断合并 |
| **门禁3: 安全扫描** | PR 创建 | OWASP 无 HIGH/CRITICAL + Semgrep 无 HIGH | 阻断合并 |
| **门禁4: 构建+镜像** | 合并后 | 构建成功 + Trivy 镜像扫描无 CRITICAL | 阻断发布 |
| **门禁5: 部署验证** | 部署后 | 冒烟测试通过 + 错误率 < 基线 | 自动回滚 |

### 3.2 代码审查 Checklist

- [ ] 是否有对应的单元测试 / 集成测试
- [ ] 是否遵循了领域模型（聚合根/实体/值对象）
- [ ] 是否有 SQL 注入风险（必须参数化查询）
- [ ] 敏感信息是否硬编码
- [ ] 日志中是否包含用户明文信息
- [ ] 异常处理是否完善（不吞异常、不暴露内部细节）
- [ ] API 变更是否更新了 Protobuf / OpenAPI 文档

## 五、专项测试方案

### 5.1 3D 渲染与性能测试

| 测试项 | 方法 | 工具 | 通过标准 |
|--------|------|------|---------|
| 帧率测试 | 真机自动化帧率采集 | 自研 FPS Monitor + Grafana | ≥ 30fps (中端) / ≥ 60fps (旗舰) |
| 内存测试 | 空间进入/退出内存泄漏检测 | Xcode Instruments / Android Profiler | 退出后内存恢复基线 ± 10% |
| 发热测试 | 连续使用 30分钟温度监控 | 温度传感器 + 自动化脚本 | 温度 < 45°C (常温环境) |
| 兼容性测试 | 主流机型空间渲染 | 真机农场 / 云真机 | Top 50 机型 crash rate < 0.1% |
| 视觉回归 | 同一空间在不同机型上的渲染截图对比 | Percy / 自研工具 | 结构一致性 > 95% |

### 5.2 AI 对话质量测试

| 测试维度 | 方法 | 评估标准 |
|---------|------|---------|
| 回复相关性 | 自动评估 (BERTScore) + 人工抽样 | BERTScore F1 > 0.7 |
| 角色一致性 | 虚拟人性格模板一致性评估 | 人工评估 80% 通过 |
| 安全性 | 对抗性测试集 (10000+ 违规样本) | 误拦截率 < 0.1%，漏拦截率 < 0.01% |
| 情感表达 | 5维情感输出正确性 | 单元测试 100% 通过 |
| 长期记忆 | 多轮跨会话记忆一致性 | 自动化脚本 100% 召回 |

### 5.3 网络与弱网测试

| 网络条件 | 带宽 | 延迟 | 丢包率 | 测试场景 |

## 六、质量度量体系

### 6.1 核心质量指标

| 指标类别 | 指标名称 | 计算方式 | 目标值 |
|---------|---------|---------|--------|
| **覆盖率** | 行覆盖率 | 覆盖行数 / 总行数 | ≥ 80% |
| | 分支覆盖率 | 覆盖分支 / 总分支 | ≥ 70% |
| **缺陷** | 缺陷密度 | Bug 数 / 千行代码 | < 2.0 |
| | 缺陷逃逸率 | 生产发现 / 总缺陷数 | < 5% |
| | P0 Bug 数 (月) | — | = 0 |
| **效率** | MTTR (平均修复时间) | 总修复时间 / 缺陷数 | < 4小时 (P0) |
| | 测试执行时间 | 全量回归耗时 | < 30分钟 |
| **可靠性** | Crash Rate | 崩溃次数 / 启动次数 | < 0.1% |
| | API 可用性 | (总请求-5xx) / 总请求 | > 99.9% |

## 七、测试环境管理

### 7.1 环境矩阵

| 环境 | 用途 | 数据 | 部署方式 | 访问 |
|------|------|------|---------|------|
| **Dev** | 开发自测 | Mock + 少量测试数据 | 自动 (Git Push) | 开发团队 |

| **Production** | 线上 | 真实数据 | 审批+灰度 | 真实用户 |

### 7.2 测试数据管理

| 策略 | 说明 |
|------|------|
| 数据工厂 | 按实体类型提供 Builder 模式创建测试数据 |
| 数据清理 | 每个测试类执行后自动回滚/清理 |
| 敏感数据 | CI/Dev 环境完全脱敏，不得包含真实 PII |
| Staging 同步 | 每周从生产脱敏同步，保持数据分布一致性 |

| 性能测试 | k6 + JMeter | 压力/负载测试 |

### 8.2 质量改进循环 (PDCA)

```
Plan (计划)
├── 设定质量目标（覆盖率/缺陷密度/MTTR）
├── 识别质量薄弱环节（故障复盘/逃逸分析）
└── 制定改进措施

Do (执行)
├── 实施改进措施
├── 更新测试用例/自动化脚本
└── 培训/知识分享

Check (检查)
├── 质量指标趋势监控
├── 对比目标与实际的差距
└── 自动化巡检报告

Act (改进)
├── 固化有效措施到流程/工具
├── 修正无效措施
└── 进入下一个循环
```
